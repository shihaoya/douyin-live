# 抖音直播监控系统 - 重构需求文档 v2.0

## 📋 项目背景

### 现状问题
1. **依赖外部程序**：当前系统依赖 `douyinLive-windows-amd64.exe`（Go语言编写），增加了部署复杂度
2. **数据混乱**：消息类型映射不清晰，各种消息混杂在一起
3. **前端耦合**：前后端未分离，不利于独立部署和扩展
4. **性能瓶颈**：大数据量下查询慢，缺少合理的缓存和索引策略

### 重构目标
构建一个**完全自主可控**的抖音直播监控系统：
- ✅ 纯Node.js实现，不依赖外部程序
- ✅ 清晰的消息分类和处理机制
- ✅ 前后端分离架构
- ✅ 高性能数据存储和查询
- ✅ 模块化设计，易于扩展

**目标**：参考GitHub项目https://github.com/jwwsjlm/douyinLive，用Node.js实现抖音直播消息监听

**技术要点**：

**关键挑战**：

#### 1.2 消息存储模块
**目标**：高效存储原始消息和处理后的消息

**数据库设计**：

##### 表1：原始消息表（raw_messages）
```sql
CREATE TABLE raw_messages (
  id BIGINT AUTO_INCREMENT PRIMARY KEY,
  room_id VARCHAR(50) NOT NULL COMMENT '直播间ID',
  method VARCHAR(100) NOT NULL COMMENT '消息方法名（WebcastChatMessage等）',
  message_id VARCHAR(100) COMMENT '消息唯一ID',
  payload JSON NOT NULL COMMENT '完整的原始JSON数据',
  received_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP COMMENT '接收时间',
  
  INDEX idx_room_method (room_id, method),
  INDEX idx_message_id (message_id),
  INDEX idx_received_at (received_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='原始消息表';
```

**特点**：
- 只存储最原始的数据，不做任何处理
- 用于问题排查和数据回溯
- 可以定期清理（保留7天）

##### 表2：处理后消息表（messages）
```sql
CREATE TABLE messages (
  id BIGINT AUTO_INCREMENT PRIMARY KEY,
  room_id VARCHAR(50) NOT NULL COMMENT '直播间ID',
  
  -- 消息分类
  socket_type VARCHAR(50) NOT NULL COMMENT 'Socket类型（对应method）',
  message_type VARCHAR(50) NOT NULL COMMENT '消息类型（中文类型名）',
  
  -- 用户信息
  user_id VARCHAR(100) COMMENT '用户ID',
  user_nickname VARCHAR(255) COMMENT '用户昵称',
  user_level INT COMMENT '用户等级',
  fans_level INT COMMENT '粉丝团等级',
  
  -- 消息内容
  content TEXT COMMENT '消息内容（弹幕文本等）',
  
  -- 礼物相关
  gift_id INT COMMENT '礼物ID',
  gift_name VARCHAR(255) COMMENT '礼物名称',
  gift_count INT DEFAULT 0 COMMENT '礼物数量',
  gift_diamond_value INT DEFAULT 0 COMMENT '礼物抖币价值',
  combo_count INT DEFAULT 1 COMMENT '连击次数',
  group_id VARCHAR(100) COMMENT '连击组ID（用于去重）',
  
  -- 其他结构化字段
  extra_data JSON COMMENT '额外的结构化数据',
  description TEXT COMMENT '可读的描述文本（根据配置动态生成）',
  
  -- 元数据
  message_id VARCHAR(100) COMMENT '消息唯一ID',
  received_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP COMMENT '接收时间',
  
  -- 索引
  INDEX idx_room_socket (room_id, socket_type),
  INDEX idx_room_message (room_id, message_type),
  INDEX idx_user (user_id),
  INDEX idx_gift (gift_id),
  INDEX idx_received_at (received_at),
  UNIQUE INDEX idx_message_id (message_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='处理后消息表';
```

**特点**：
- 结构化的数据，便于查询和统计
- 通过唯一索引实现去重
- 支持多维度筛选

#### 1.3 消息处理引擎
**目标**：根据配置动态解析消息并生成描述

**工作流程**：
```
1. 接收原始消息
2. 根据 method 查找对应的解析配置
3. 使用 description_template 生成描述文本
4. 保存到 messages 表（包含 description 字段）
5. 同时保存原始数据到 raw_messages 表
6. 如果解析失败，description 设置为 "异常"
```

**描述模板语法**：

##### 基本语法
```
// 单一路径
user.nickname

// 多路径回退（用 || 分隔，从左到右尝试）
user.nickname || user.name || "未知用户"

// 嵌套路径
gift.name

// 深层嵌套
gift.extra.info.name

// 混合使用
user.nickname + " 赠送了 " + gift.name + " x" + gift.count
```

##### 错误处理机制

```javascript
// 1. 语法错误（模板写错了）
description_template: "user..nickname"  // 双点号
→ 返回 "异常"，记录错误日志

// 2. 运行时错误（数据结构不符合预期）
data: { user: null }
template: "user.nickname"
→ 返回 "异常"，记录错误日志

// 3. 部分字段缺失（有回退机制）
data: { user: { name: "李四" } }
template: "user.nickname || user.name"
→ 返回 "李四"（成功回退）

// 4. 所有回退都失败
data: { user: {} }
template: "user.nickname || user.name || '未知'"
→ 返回 "未知"（使用默认值）
```

## ⚠️ 关键风险与挑战

### 1. 抖音WebSocket协议解析
**风险等级**：🔴 高  
**影响**：无法获取直播消息，整个项目失败  
**应对方案**：
- 提前调研可行性
- 准备多个技术方案
- 如果太难，考虑继续使用douyinLive作为底层服务

### 2. 高并发写入性能
**风险等级**：🟡 中  
**影响**：消息丢失或延迟  
**应对方案**：
- 使用批量插入
- 优化数据库索引
- 考虑消息队列缓冲（RabbitMQ/Kafka）

### 3. 数据一致性
**风险等级**：🟡 中  
**影响**：统计数据不准确  
**应对方案**：
- 使用事务保证原子性
- 定期数据校验
- 提供数据修复工具
